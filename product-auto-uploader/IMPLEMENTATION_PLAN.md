# 자동 업로드 MVP 구현 계획

참고:
이 문서는 초기 단일 사용자 MVP 기준 초안이다.
현재 4인 배포 기준 최신 방향은 [DEPLOYMENT_PLAN.md](C:\Users\freetime\Desktop\python-projects\product-auto-uploader\DEPLOYMENT_PLAN.md)를 우선 본다.

## 1. 목표 재정의

현재 단계의 목표는 "머스트잇 상품 등록 자동화가 최소한으로 동작하는 로컬 도구"를 만드는 것이다.

사용자는 다음 정보만 입력한다.

- 카테고리
- 브랜드명
- 상품번호
- 제품명
- 가격

도구는 다음만 수행한다.

- 로컬 사진 폴더를 찾는다
- 머스트잇 상품 등록 페이지를 연다
- 정해진 필드를 자동 입력한다
- 사진을 업로드한다
- 최종 등록 직전 또는 등록 완료 직후까지 진행한다

이번 단계에서는 "동작하는 것"을 우선하고, 범용성이나 고급 편의 기능은 뒤로 미룬다.

## 2. 전제와 가정

사용자와 추가 확인이 어렵기 때문에 아래를 기본 가정으로 둔다.

- 실행 환경은 Windows 로컬 PC다.
- 사용자는 머스트잇 계정을 이미 가지고 있다.
- 로그인은 자동화하지 않고, 최초 1회 또는 필요 시 사용자가 직접 로그인한다.
- 상품 등록에 꼭 필요한 추가 필드가 있더라도, 우선은 고정값 또는 빈값 처리 가능한 범위만 자동화한다.
- 사진 경로는 `C:\Users\freetime\Desktop\registerPic\<brand>\<product_code>` 형식이다.
- 브랜드명은 입력값을 소문자/공백 제거 등 최소 정규화해서 폴더명에 매핑한다.
- 사이트 셀렉터 변경 가능성이 높으므로, 셀렉터 정의는 코드에서 분리한다.

이 가정이 틀리면 구현은 가능해도 바로 실사용은 어려울 수 있다. 그래서 MVP는 "한 PC, 한 사용자, 한 사이트"에 최적화한다.

## 3. 권장 기술 선택

최소 구현 기준으로 아래 조합을 권장한다.

- 언어: Python 3.11+
- 브라우저 자동화: Playwright
- 입력 UI: Streamlit 또는 간단한 CLI
- 설정 관리: `.env` + `config.json`
- 데이터 검증: Pydantic
- 로깅: 표준 `logging`

### 왜 이 조합이 맞는가

- Playwright는 파일 업로드와 대기 처리, 셀렉터 안정성이 Selenium보다 편하다.
- Streamlit은 빠르게 입력 폼을 만들 수 있다.
- 다만 정말 최소 구현만 원하면 1차는 CLI로 시작하고 2차에 Streamlit을 붙이는 것이 더 빠르다.

실행 속도를 우선하면 아래 순서를 추천한다.

1. 1차: CLI 기반 등록 자동화
2. 2차: 같은 내부 로직에 Streamlit 폼만 추가

## 4. MVP 범위

### 포함

- 머스트잇 1개 사이트 자동 등록
- 상품 기본 정보 5개 입력
- 사진 폴더 자동 탐색
- 다중 이미지 업로드
- 실행 로그 저장
- 실패 시 중단 지점과 사유 표시

### 제외

- 필웨이, 트렌비 연동
- 공식 홈페이지 크롤링
- 재고, 옵션, 상세설명 자동 생성
- 다계정 지원
- 대량 일괄 업로드
- CAPTCHA, 2FA, 보안 우회
- 완전 무인 로그인

## 5. 구현 방식

핵심은 "입력", "검증", "자동화", "결과 기록"을 분리하는 것이다.

### 5.1 사용자 입력 계층

입력은 내부적으로 하나의 `ProductInput` 모델로 통일한다.

필드 예시:

- `category`
- `brand_name`
- `product_code`
- `product_name`
- `price`

이렇게 해두면 나중에 Streamlit, CLI, CSV 업로드가 모두 같은 로직을 재사용할 수 있다.

### 5.2 로컬 파일 해석 계층

입력값으로 사진 폴더를 찾는다.

처리 규칙:

- 브랜드명을 폴더명 규칙으로 정규화
- `registerPic/<brand>/<product_code>` 경로 생성
- 이미지 파일만 필터링
- 파일이 없으면 자동화 시작 전에 실패 처리

지원 확장자는 우선 아래 정도면 충분하다.

- `.jpg`
- `.jpeg`
- `.png`
- `.webp`

### 5.3 사이트 자동화 계층

머스트잇 전용 업로더 클래스를 둔다.

예시:

- `BaseUploader`
- `MustitUploader`

`MustitUploader` 책임:

- 브라우저 실행
- 로그인 세션 확인
- 등록 페이지 이동
- 각 필드 입력
- 사진 업로드
- 제출 또는 제출 직전 대기

사이트별 차이는 셀렉터와 입력 절차에 몰아넣는다.

### 5.4 실행 제어 계층

`UploadService`가 전체 흐름을 오케스트레이션한다.

순서:

1. 입력값 검증
2. 사진 경로 검증
3. 브라우저 실행
4. 업로더 호출
5. 결과 저장

### 5.5 결과 기록 계층

실행 결과는 JSON 또는 CSV로 남긴다.

최소 저장 항목:

- 실행 시각
- 사이트명
- 상품번호
- 성공 여부
- 실패 사유
- 스크린샷 경로

## 7. 단계별 구현 계획

### 1단계: 기본 골격

목표:

- 프로젝트 실행 가능 상태 만들기
- 입력 모델과 설정 파일 구조 확정

할 일:

- Python 가상환경 생성
- `requirements.txt` 작성
- Playwright 설치
- `ProductInput` 모델 정의
- 설정값 로딩 구조 정의

완료 기준:

- 샘플 입력값으로 프로그램이 시작된다.

### 2단계: 사진 탐색 로직

목표:

- 브랜드명과 상품번호만으로 사진 경로를 확정한다

할 일:

- 브랜드명 정규화 함수 작성
- 폴더 존재 여부 검사
- 이미지 목록 수집
- 예외 메시지 정리

완료 기준:

- 잘못된 경로와 빈 폴더를 명확히 구분해 실패시킨다.

### 3단계: 머스트잇 수동 보조 자동화

목표:

- 사용자가 로그인한 상태에서 등록 페이지까지 도달하고 기본 입력이 된다

할 일:

- Playwright persistent context 사용
- 사용자 프로필 또는 세션 저장 경로 설정
- 머스트잇 등록 페이지 이동
- 주요 필드 셀렉터 파악
- 텍스트 필드 자동 입력

완료 기준:

- 카테고리 제외 텍스트 필드 입력이 안정적으로 동작한다.

### 4단계: 사진 업로드 연결

목표:

- 로컬 이미지가 실제 등록 화면에 업로드된다

할 일:

- 업로드 input 셀렉터 연결
- 다중 파일 업로드 처리
- 업로드 완료 대기 조건 설정
- 업로드 실패 시 스크린샷 저장

완료 기준:

- 지정 폴더의 이미지가 화면에 반영된다.

### 5단계: 카테고리 입력 안정화

목표:

- 카테고리 선택이 최소 케이스에서 동작한다

권장 방식:

- 1차는 카테고리 자유 입력 또는 단순 매핑이 가능하면 그 방식 사용
- 드롭다운 다단 선택만 가능하면, 자주 쓰는 소수 카테고리만 하드코딩 매핑

이유:

- 카테고리 UI는 사이트에서 가장 자주 깨지는 부분이라 전체 범용화를 나중으로 미루는 것이 맞다.

완료 기준:

- 최소 1~3개 실제 카테고리에서 재현 가능하게 동작한다.

### 6단계: 제출 정책 결정

목표:

- 실수 방지를 위해 기본 동작을 보수적으로 잡는다

권장 기본값:

- 기본 모드는 "제출 직전까지 자동 입력 후 대기"
- 옵션으로만 "자동 제출" 허용

이유:

- 사용자 확인 없이 완전 자동 제출은 초기 리스크가 크다.

완료 기준:

- 사용자가 최종 등록 버튼만 직접 누르면 되는 상태까지 자동화된다.

### 7단계: 로그와 실패 복구

목표:

- 실패 원인을 다시 추적할 수 있게 만든다

할 일:

- 단계별 로그
- 실패 시 스크린샷
- 마지막 처리 상품 정보 저장

완료 기준:

- 실패해도 어디서 실패했는지 바로 파악된다.

## 8. 첫 구현에서 가장 중요한 절충안

### 로그인

자동 로그인보다 "persistent browser profile"을 쓰는 편이 낫다.

- 사용자가 한번 로그인
- 이후 같은 프로필을 재사용
- 세션이 풀리면 다시 로그인

이 방식이 가장 단순하고 유지보수 비용이 낮다.

### 카테고리

전체 카테고리 자동 매핑은 미루는 것이 맞다.

- 1차는 제한된 카테고리만 지원
- 카테고리 문자열과 사이트 선택값을 매핑 테이블로 관리

### 제출

초기에는 자동 제출을 기본으로 두지 않는다.

- 실수로 잘못 등록될 비용이 큼
- 사용자는 마지막 화면만 확인하면 됨

## 9. 리스크와 대응

### 사이트 구조 변경

대응:

- 셀렉터를 `selectors/mustit.json`으로 분리
- 코드 수정 없이 셀렉터만 교체 가능하게 구성

### 필수 입력항목 누락

대응:

- MVP 실행 전에 실제 등록 화면을 한 번 수동으로 끝까지 진행해 필수 필드를 파악
- PRD에 없는 필드는 고정값 처리 가능한지 먼저 확인

### 이미지 폴더명 불일치

대응:

- 브랜드명 정규화 규칙 추가
- 필요 시 `brand_alias_map` 설정 파일 제공

### 업로드 속도/타임아웃

대응:

- 업로드 완료 대기시간을 설정값으로 분리
- 실패 시 재시도 1회만 허용

## 10. 바로 착수 가능한 개발 순서

가장 현실적인 착수 순서는 아래다.

1. CLI 입력 기반으로 `ProductInput` 받기
2. 사진 폴더 탐색 로직 구현
3. Playwright persistent login 구조 구현
4. 머스트잇 등록 페이지에서 텍스트 필드 입력 구현
5. 이미지 업로드 구현
6. 제출 직전 대기 모드 구현
7. 로그와 스크린샷 저장 추가

이 순서면 중간에 UI를 바꾸더라도 핵심 자동화 코드는 유지된다.

## 11. 추후 확장 방식

필웨이, 트렌비 확장은 같은 인터페이스를 추가하는 방식으로 간다.

- `MustitUploader`
- `FeelwayUploader`
- `TrenbeUploader`

공통 입력 모델과 이미지 탐색 로직은 재사용한다.

나중에 공식 홈페이지 크롤링을 붙일 때도 분리한다.

- `ProductEnrichmentService`
- 상품번호 기반 데이터 보강

즉, "입력 수집", "로컬 자산 조회", "사이트 업로드", "외부 데이터 보강"을 처음부터 분리해두는 것이 핵심이다.

## 12. 최종 권장안

현재 상황에서는 아래가 가장 적절하다.

- 1차 구현은 Python + Playwright + CLI
- 머스트잇만 지원
- 로그인은 수동, 등록 입력은 자동
- 사진은 로컬 경로 규칙 기반 조회
- 기본 모드는 제출 직전 대기

그 다음 단계에서 붙일 것:

- Streamlit 입력 폼
- 카테고리 매핑 고도화
- 필웨이/트렌비 업로더 추가
- 공식 홈페이지 크롤링

이렇게 하면 초기 구현 리스크를 가장 낮추면서도, 나중에 구조를 갈아엎지 않고 확장할 수 있다.
